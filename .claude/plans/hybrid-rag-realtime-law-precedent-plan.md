> 생성일: 2025-12-13  
> 업데이트: 2025-12-13  
> 목표: **RAG DB(스토어)에 없는 법령/판례도 실시간 API 호출로 근거를 끌고 와서** AI 답변에 반영 (n8n처럼 “필요할 때만 도구 호출”)

## 1) 결론(질문 답변)

가능합니다. 구현 방식은 크게 2가지입니다.

1. **(권장) 서버 오케스트레이터 방식**: 먼저 File Search RAG로 답변을 시도하고, **인용(citation) 부족/미존재 시에만** 법령/판례 API를 호출해 “근거 패킷(evidence pack)”을 만든 뒤, 그 근거로 다시 답변 생성.
2. **모델 Tool Calling 방식**: Gemini에 `search_law`, `get_law`, `search_precedent`, `get_precedent` 같은 도구를 선언하고, 모델이 필요 시 호출 → 결과를 근거로 답변.

LexDiff는 이미 **SSE 스트리밍 + 인용 기반 신뢰도** 흐름이 있으므로, 초기 구현은 (1)이 빠르고 안정적입니다.

## 2) 현재 LexDiff 기반(현 상태) 정리

- RAG 엔드포인트: `app/api/file-search-rag/route.ts` (SSE, `citations`, `confidenceLevel`, `queryType` 전송)
- 질문 분석: `lib/legal-query-analyzer.ts` (질문 유형 + 법령/조문 추출)
- 법제처 API 연동: 프로젝트 전반에 존재 (예: `/api/eflaw` 등)
- 심판례(샘플) 검색/다운로드 가이드: `docs/18-precedent-search-implementation-guide.md`
  - `kcsCgmExpc` (관세청 법령해석), `ttSpecialDecc` (조세심판원)
- 판례(공동활용/DRF): **판례 검색/조회 타겟을 우선 사용**(예: `target=prec` 계열) → “온디맨드 검색+근거화”에 적합  
  - ⚠️ 응답 필드/전문 제공 범위는 DRF 문서 기준으로 확정(구현 전 스펙 확인)
- 비용 기준(레포 문서): `docs/06-GEMINI_FILE_SEARCH_GUIDE.md`
  - 답변 생성: **Gemini 2.5 Flash $0.075(입력) / $0.30(출력) per 1M tokens**

## 3) 목표 아키텍처(하이브리드)

### 3.1 흐름 요약

1. 사용자 질문 → `analyzeLegalQuery()`로 질문 유형/법령/조문 후보 추출
2. File Search RAG 질의 → `citations` / `avg relevanceScore` 확인
3. 조건 충족 시 **실시간 근거 수집**(법령/판례)
4. 근거 패킷(evidence pack) + 기존 RAG 인용을 합쳐 **최종 답변 생성**
5. UI에는 “근거(법령/판례)”를 분리 표기하고, 링크/식별자 제공

### 3.2 의사결정 규칙(권장, 단순/강건)

| 상황 | 처리 |
|---|---|
| `citations.length >= 1` AND `avgScore >= 0.4` | **File Search 결과만으로 답변** (추가 API 호출 없음) |
| `citations.length == 0` | **법령/판례 실시간 호출 트리거** (환각 방지) |
| 질문 유형이 `application`/`consequence` + 사실관계가 구체적 | 판례 검색을 “옵션”이 아니라 “우선”으로(상위 N개) |
| 조문 번호가 명시됨(예: “제38조”) | 법령 API는 “해당 조문만” 가져오기(토큰 절감) |

## 4) 실시간 “근거 패킷” 설계

### 4.1 공통 스키마(서버 내부)

```ts
type Evidence =
  | {
      kind: 'statute'
      source: 'law.go.kr'
      title: string          // 예: 관세법 제38조
      lawName: string
      article?: string
      url?: string
      excerpt: string        // 인용할 최소 발췌(필수)
      fetchedAt: string
    }
  | {
      kind: 'precedent'
      source: 'law.go.kr-prec' | 'kcs' | 'tt' | 'vendor'
      title: string
      id: string
      date?: string
      url?: string
      summary: string        // 요약(필수)
      keyHoldings?: string[] // 쟁점별 포인트(선택)
      fetchedAt: string
    }
```

### 4.2 토큰/품질 가드레일

- 법령: **최대 3개 조문**, 각 조문 발췌 **최대 1,200자**
- 판례/심판례: 검색 결과 **상위 3~5개만**, 각 요약 **최대 1,200자**
- “근거 없는 일반 지식 답변”을 금지: 근거가 없으면 “근거 부족”으로 응답하고 추가 질문 유도

## 5) 법령(스토어에 없을 때만 호출) 구현 계획

### 5.1 법령 누락 감지

- 1차: File Search 결과 `citations.length == 0`
- 2차(정교화): `analyzeLegalQuery()`의 `extractedLaws`가 있는데도 citations가 없으면 “법령 API 호출”

### 5.2 호출 전략

- 조문이 있으면: `extractedArticles` 기준으로 **해당 조문만** 조회(가장 저렴/정확)
- 조문이 없으면: 관련 조문 후보를 좁히기 위해
  - (빠른 버전) 법령 전체 중 “정의/요건/벌칙/신고/면허” 섹션만 우선 발췌
  - (정확 버전) 한 번 더 모델 호출(저토큰)로 “필요 조문 리스트”만 뽑고, 그 조문만 API로 가져오기

### 5.3 캐시

- 서버 캐시(권장): Turso/LibSQL에 `lawName+article+date` 키로 7~30일 TTL
- 브라우저 캐시(보조): 기존 IndexedDB 캐시 정책과 정합(필요 시)

## 6) 판례/심판례(온디맨드 검색+요약) 구현 계획

### 6.1 1차 범위(빠르게 가능, “공동활용 판례 + 기존 심판례/해석례”)

`docs/18-precedent-search-implementation-guide.md` 기반으로 아래를 “AI 답변용 도구”로 재사용:

- 판례: **공동활용 DRF 판례 검색/상세(예: `target=prec` 계열)** → 상위 N개만 근거화
- 해석례/심판례: `kcsCgmExpc`, `ttSpecialDecc` (law.go.kr DRF)
- 상세: 선택한 ID로 전문 조회 → Markdown으로 정리

### 6.2 공동활용 판례를 “정식 판례”로 쓰는 방식(권장)

사용자 말대로 공동활용에서 판례가 검색된다면, 초기 제품은 이 루트를 주력으로 잡는 게 좋습니다.

- 검색(예시): `GET http(s)://www.law.go.kr/DRF/lawSearch.do?target=prec&OC=...&type=JSON&query=...&display=...`
- 상세(예시): `GET http(s)://www.law.go.kr/DRF/lawService.do?target=prec&OC=...&type=JSON&ID=...`
- 근거화 포인트(권장): 전문 전체를 넣기보다, DRF가 주는 **판시사항/판결요지/참조조문/참조판례**를 우선 사용하고 필요 시에만 추가 발췌

⚠️ 정확한 파라미터/필드명/전문 제공 여부는 DRF 스펙을 확인해 확정합니다(구현 단계에서 샘플 응답으로 타입 고정).

### 6.3 (보완) 공동활용으로 커버되지 않는 판례 소스 확장

공동활용 판례의 커버리지/전문 제공이 부족한 경우 대비용 옵션:

1. **공식/제공 API가 있는 소스**(우선)로 연결 (식별자/URL/전문 확보)
2. **라이선스가 있는 상용 DB** 연동 (가장 안정적, 비용/계약 필요)
3. **스크래핑**(비권장): 법적/기술적 리스크가 커서 제품화 단계에서는 피하는 게 안전

### 6.4 요약 방식(환각 방지)

- “요약”은 단독 생성이 아니라, **전문(또는 핵심 부분) + 구조화 프롬프트**로 생성
- 출력 템플릿(권장):
  - 사건 개요(한 줄)
  - 쟁점(불릿 2~4)
  - 판단/결론(불릿 2~4)
  - 본 사안 적용 시 체크포인트(불릿)
  - 출처(식별자/링크)

## 8) 비용 재산정(2.5 Flash 기준)

레포 기준(`docs/06-GEMINI_FILE_SEARCH_GUIDE.md`):

- 입력 토큰: **$0.075 / 1M tokens**
- 출력 토큰: **$0.30 / 1M tokens**

### 8.1 쿼리당 비용(근사)

가정(권장 기본값):
- 입력: 4,000 tokens (시스템/질문/근거 패킷 포함)
- 출력: 1,200 tokens

그러면 1회 답변 생성 비용:
- 입력: `4000/1e6 * 0.075 = $0.000300`
- 출력: `1200/1e6 * 0.30  = $0.000360`
- 합계: **$0.000660 / query**

### 8.2 월 비용 예시(100 사용자/일)

| 시나리오 | 1인당 1일 질의 | 월 질의(30일) | 월 비용(근사) |
|---|---:|---:|---:|
| 기본(법령 위주) | 1 | 3,000 | ~$1.98 |
| 활발(평균 2회) | 2 | 6,000 | ~$3.96 |
| 헤비(평균 5회) | 5 | 15,000 | ~$9.90 |

### 8.3 판례 요약을 “별도 호출”로 할 때

판례 요약을 답변과 분리해서 1회 더 호출하면(예: 입력 2,000 / 출력 500):
- 추가 비용: `2000/1e6*0.075 + 500/1e6*0.30 = $0.000150 + $0.000150 = $0.000300`
- 월 3,000건 모두 판례 요약을 한다면: **+$0.90/월**

→ 결론: **판례를 온디맨드로 붙여도 비용은 매우 낮은 편**이고, “근거 품질”이 더 큰 변수입니다.

## 9) 구현 로드맵(2주 플랜, 현실적인 범위)

### Week 1 — 하이브리드 파이프라인 MVP

1. `citations==0`일 때만 동작하는 “근거 수집” 추가

**완료 기준**: “스토어에 인용이 없을 때도” 법령/심판례 근거를 붙여 답변이 생성되고, 출처 링크/ID가 함께 출력됨.

### Week 2 — 품질/UX/캐시/평가

1. Evidence 캐시(Turso) + 재사용
