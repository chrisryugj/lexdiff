# LexDiff 향후 발전 로드맵 (Phase 8+)

## 현재 상태 (2025-11-11)

### ✅ 활성화된 시스템
- **Phase 7**: IndexedDB 캐시 (~25ms, 매우 빠름)
- **기본 검색**: law-search API + 레벤슈타인 유사도 매칭
- **장점**: 속도 ⚡, 정확도 ✅

### ❌ 비활성화된 시스템
- **Phase 5**: Intelligent Search (학습 데이터 오염)
- **Phase 6**: Vector Search (Phase 5와 함께 비활성화)

### 🔍 현재 한계점
1. **자연어 검색 불가**: "세금 관련 법령" → 처리 불가
2. **오타 처리 제한적**: search-normalizer가 몇 가지만 처리 (벚→법, 셰→세)
3. **동의어 미지원**: "택시" vs "택시운송사업" 등
4. **맥락 이해 부족**: "수입품에 대한 세금" → "관세법" 연결 불가

## Phase 8: 스마트 오타 교정 시스템 (1-2주)

### 개념
사용자 입력의 오타를 자동으로 감지하고 수정하여 검색 성공률 향상

### 구현 방법

#### Option A: 규칙 기반 오타 교정 (빠른 구현)
```typescript
// lib/typo-corrector.ts

// 1. 한글 자소 분리 오타 (예: "관셰법" → "관세법")
// 2. 자판 인접 오타 (예: "관새법" → "관세법")
// 3. 받침 누락/추가 (예: "관세벚" → "관세법")
```

**장점**:
- 빠른 실행 속도 (~1ms)
- API 비용 없음
- 오프라인 작동

**단점**:
- 패턴 기반으로 한계 있음
- 새로운 오타 패턴 수동 추가 필요

#### Option B: AI 기반 오타 교정 (추천) ⭐
```typescript
// lib/ai-typo-corrector.ts

async function correctTypo(query: string): Promise<{
  original: string
  corrected: string
  confidence: number
  suggestions: string[]
}> {
  // Gemini Flash로 오타 교정 요청
  const prompt = `
  다음 법령 검색어의 오타를 수정해주세요:
  입력: "${query}"

  한국 법령명 목록 (참고):
  - 형법, 민법, 상법, 관세법, 소득세법, 법인세법...

  JSON 응답:
  {
    "corrected": "수정된 검색어",
    "confidence": 0.95,
    "suggestions": ["대안1", "대안2"]
  }
  `
}
```

**장점**:
- 맥락 기반 수정 (예: "관새벚" → "관세법")
- 새로운 패턴 자동 학습
- 동의어도 처리 가능

**단점**:
- API 비용 (Flash는 저렴: ~$0.000075/요청)
- ~200-300ms 지연

#### Option C: 하이브리드 (최적) ⭐⭐
```typescript
// 1단계: 규칙 기반 빠른 교정 (1ms)
// 2단계: 규칙으로 해결 안 되면 AI 교정 (200ms)
```

### 구현 단계
1. **기본 오타 사전 구축** (1일)
   - 자주 발생하는 오타 패턴 수집
   - 한글 자소 분리 로직
   - 자판 인접 키 매핑

2. **Gemini Flash 통합** (2일)
   - `/api/correct-typo` 엔드포인트
   - 캐싱 (동일 오타는 재사용)
   - 타임아웃 처리 (300ms 초과 시 원본 사용)

3. **UI 통합** (1일)
   - "혹시 '관세법'을 찾으시나요?" 제안
   - 자동 수정 + 되돌리기 버튼

### 예상 효과
- 검색 성공률 **+15~20%**
- 사용자 재검색 감소

## Phase 9: 자연어 → 법령명 변환 (2-3주) ⭐⭐⭐

### 개념
사용자가 자연어로 질문하면 관련 법령을 찾아주는 시스템

### 예시
```
사용자: "수입품에 대한 세금은 어떤 법에?"
→ AI 분석: "관세법"
→ 검색 실행: 관세법 표시

사용자: "택시 운전사 처벌 규정"
→ AI 분석: "여객자동차 운수사업법" + "형법"
→ 2개 법령 제안
```

### 구현 방법

#### Architecture
```typescript
// lib/natural-language-search.ts

interface NLSearchResult {
  intent: 'law-search' | 'article-search' | 'keyword-search'
  lawNames: string[]        // 추천 법령명들
  keywords: string[]        // 핵심 키워드
  confidence: number        // 0~1
  reasoning: string         // AI의 추론 과정
}

async function analyzeNaturalLanguageQuery(
  query: string
): Promise<NLSearchResult> {
  // Gemini 2.5 Flash로 분석
  const prompt = `
  사용자 질문을 분석하여 관련 법령을 찾아주세요.

  질문: "${query}"

  한국 주요 법령 목록:
  - 형법 (범죄, 처벌)
  - 민법 (계약, 재산, 가족)
  - 상법 (회사, 상거래)
  - 관세법 (수입, 수출, 세금)
  - 소득세법 (소득, 세금)
  - 여객자동차 운수사업법 (택시, 버스)
  - 전기통신사업법 (통신, 인터넷)
  ...전체 300+ 법령

  응답 형식 (JSON):
  {
    "intent": "law-search",
    "lawNames": ["관세법", "대외무역법"],
    "keywords": ["수입품", "세금"],
    "confidence": 0.95,
    "reasoning": "수입품 관련 세금은 관세법에서 규정"
  }
  `

  return parseGeminiResponse(response)
}
```

#### 법령 목록 제공 방식

**Option A: Static 법령 목록** (추천)
- 300+ 주요 법령명을 프롬프트에 포함
- Gemini가 선택하도록
- 장점: 정확도 높음
- 단점: 프롬프트 길이 증가 (~2KB)

**Option B: 벡터 검색 먼저**
- 사용자 질문 → 임베딩
- 유사한 법령 10개 찾기
- Gemini가 최종 선택
- 장점: 프롬프트 짧음
- 단점: Phase 6 필요

#### UI/UX Flow

```
┌─────────────────────────────────────┐
│ 검색창                               │
│ "수입품에 대한 세금은?"             │
└─────────────────────────────────────┘
           ↓
┌─────────────────────────────────────┐
│ 🤖 AI 분석 중... (300ms)            │
└─────────────────────────────────────┘
           ↓
┌─────────────────────────────────────┐
│ 💡 다음 법령을 찾으셨나요?          │
│                                     │
│ ✅ 관세법 (추천도: 95%)             │
│    └ "수입품 세금은 관세법에서 규정"│
│                                     │
│ ○ 대외무역법 (추천도: 60%)          │
│                                     │
│ [관세법 보기]  [직접 검색]          │
└─────────────────────────────────────┘
```

### 구현 단계

1. **법령 목록 DB 구축** (1일)
   - 전체 법령명 + 카테고리 + 설명
   - JSON 파일로 관리
   - 예: `law-catalog.json`

2. **Gemini API 통합** (2일)
   - `/api/analyze-nl-query` 엔드포인트
   - 프롬프트 엔지니어링
   - 응답 파싱 및 검증

3. **캐싱 시스템** (1일)
   - 동일 질문은 캐시에서 즉시 응답
   - IndexedDB 또는 Turso DB

4. **UI 구현** (2일)
   - 자연어 감지 (물음표, 조사 포함 시)
   - AI 제안 카드
   - 클릭 한 번에 법령 연결

5. **학습 시스템** (2일)
   - 사용자가 선택한 법령 기록
   - 다음번 동일 질문 시 더 정확한 추천

### 비용 추정
- Gemini 2.5 Flash: $0.075 / 1M 입력 토큰
- 평균 쿼리: ~500 토큰 (법령 목록 포함)
- 100 쿼리/일 → **$0.004/일** (매우 저렴!)

### 예상 효과
- 초보 사용자 접근성 **대폭 향상**
- "법령명을 모르는 사용자"도 검색 가능
- 전문가도 시간 절약 (예: "공정거래 관련 법" → 즉시 추천)

## Phase 5v2 & 6v2: 학습 시스템 재설계 (1개월)

### Phase 5v2: 신뢰도 기반 Intelligent Search

#### 문제점 (기존)
- 잘못된 매핑이 학습됨
- 검증 시스템 없음

#### 해결책
```typescript
interface LearningEntry {
  query: string
  lawId: string
  confidence: number      // 0~1
  verificationCount: number
  successRate: number     // 실제 사용자가 선택한 비율
  autoApply: boolean      // 자동 적용 여부
}

// 규칙:
// - confidence < 0.8: 제안만
// - confidence >= 0.8 && successRate > 90%: 자동 적용
// - successRate < 50%: 학습 데이터 삭제
```

#### 자동 검증
```typescript
// 사용자가 다른 법령 선택 → confidence 감소
// 사용자가 제안 수락 → confidence 증가
// 10회 이상 검증 → 신뢰할 수 있는 매핑
```

### Phase 6v2: 향상된 벡터 검색

#### 개선 사항
1. **더 나은 임베딩 모델**
   - Voyage AI Large (현재 사용)
   - 또는 OpenAI text-embedding-3-large

2. **법령별 임베딩 사전 생성**
   - 300+ 법령의 설명을 미리 임베딩
   - 검색 시 코사인 유사도 계산

3. **하이브리드 검색**
   - 키워드 매칭 70% + 벡터 유사도 30%
