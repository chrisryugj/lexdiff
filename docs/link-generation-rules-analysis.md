# 링크 생성 규칙 충돌 분석

## 현재 발견된 충돌 패턴

### 1. 「법령명」 vs 일반 법령명 충돌
**문제**:
```
「국토의 계획 및 이용에 관한 법률 시행령」 제61조
↓
1차 처리: 「」 안의 텍스트 → 마커 생성
2차 처리: 마커 안의 "토의 계획 및 이용에 관한 법률 시행령 제61조" 재매칭
```

**원인**: negative lookbehind `(?<!<<<[^>]*|「)` 가 마커 내부 텍스트를 완전히 보호하지 못함

### 2. 복합 법령명 충돌
**문제**:
```
"국토의 계획 및 이용에 관한 법률 시행령"
↓
1. "법률 시행령" 패턴 매칭
2. "시행령" 단독 패턴 매칭
```

**원인**: 여러 패턴이 같은 텍스트의 일부를 중복 매칭

### 3. 법령명 경계 인식 실패
**문제**:
```
"건축법 제2조제1항제2호에 따른 건축물"
↓
"건축법 제2조" + "제1항제2호" 분리
```

**원인**: 조문 번호 뒤의 항/호를 별개 엔티티로 인식

## 개선된 규칙 체계

### 우선순위 기반 단일 패스 처리

```
우선순위 1: 「법령명」 패턴 (가장 명확)
    ├─ 「법령명」 제X조제Y항제Z호
    └─ 「법령명」 단독

우선순위 2: 법령명 + 조문 패턴
    ├─ 법령명 제X조의Y제A항제B호 (전체)
    └─ 법령명 제X조의Y (조문만)

우선순위 3: 법령명 단독
    ├─ 법률 시행령/시행규칙
    └─ 단독 법령명

우선순위 4: 제X조 단독 (현재 법령)

우선순위 5: 대통령령/시행령

우선순위 6: 부령/시행규칙
```

### 충돌 방지 규칙

1. **한 번 매칭된 텍스트는 재처리 금지**
   - 토큰화 방식으로 위치 추적
   - 처리된 영역은 건너뛰기

2. **명확한 경계 설정**
   - 법령명 끝: `(?:법|령|규칙|조례)` 뒤 한글이 아닌 경우
   - 조문 끝: 숫자 뒤 "조"까지만 (항/호는 디스플레이만)

3. **복합 법령명 우선 처리**
   - "법률 시행령" > "시행령"
   - "도로법 시행규칙" > "시행규칙"

## 테스트 케이스

### 케이스 1: 꺽쇄 + 조문
```
입력: 「국토의 계획 및 이용에 관한 법률 시행령」 제61조제1호
기대: <a data-law="국토의 계획 및 이용에 관한 법률 시행령" data-article="제61조">전체텍스트</a>
```

### 케이스 2: 복합 법령명
```
입력: 국토의 계획 및 이용에 관한 법률 시행령 제61조
기대: <a data-law="국토의 계획 및 이용에 관한 법률 시행령" data-article="제61조">전체텍스트</a>
```

### 케이스 3: 중첩된 참조
```
입력: 「건축법」 제2조제1항제2호에 따른 건축물로서 국토의 계획 및 이용에 관한 법률 시행령 제61조
기대:
- <a data-law="건축법" data-article="제2조">「건축법」 제2조제1항제2호</a>
- <a data-law="국토의 계획 및 이용에 관한 법률 시행령" data-article="제61조">국토의 계획 및 이용에 관한 법률 시행령 제61조</a>
```

### 케이스 4: 연속된 법령명
```
입력: 도로법 제61조, 건축법 제2조
기대:
- <a data-law="도로법" data-article="제61조">도로법 제61조</a>
- <a data-law="건축법" data-article="제2조">건축법 제2조</a>
```

### 케이스 5: 단독 시행령
```
입력: 대통령령 제12345호
기대: <a data-ref="related" data-kind="decree">대통령령</a> 제12345호
```

## 검증 방법

1. **단위 테스트**: 각 패턴별 독립 테스트
2. **통합 테스트**: 실제 법령 텍스트로 전체 파이프라인 테스트
3. **회귀 테스트**: 기존 정상 동작 케이스 확인
4. **엣지 케이스**: 특수 문자, 긴 법령명, 중첩 패턴
